# BACKLOG (暂不做)

> 这里记录**有价值但刻意不放进当前 MVP** 的方向。不是遗漏,是为了让 Phase 0/1/2 收口。
> 每条都写清楚:为什么暂缓、什么条件下再启动。当前任务看 `docs/IMPLEMENTATION_PLAN.md`。

## B1. 失败 → eval case → CI 质量门禁(自动闭环)

把线上抓到的一次 agent 失败一键转成 eval case,进 CI,下次 PR 自动跑、退化就挡住。

- **价值**:蓝海。可观测已饱和(~89% 团队已有),"把观测变成质量保证"还没人做透。
- **为什么暂缓**:这其实是个**独立产品方向**(对标 Braintrust),不是可视化器的功能,硬塞会让 MVP 失焦。
- **触发条件**:Phase 0/1/2 全部跑通、有真实用户在用根因定位之后,再评估是否作为独立模块/产品延伸。
- **当前影子**:Phase 2.5 的"两次 run 对比"算是最轻量的前身。

## B2. 多 agent / subgraph 编排调试

可视化"谁在等谁、竞态发生在哪、级联失败从哪开始",而不只是单条线性/单图执行。

- **价值**:多 agent 互相委派时编排复杂度近乎指数增长,这块现有工具最没做透,技术壁垒也最高。
- **为什么暂缓**:技术难度最大;当前计划只处理"单个图带 cycle"的布局与回环。先验证单图体验。
- **触发条件**:单图的根因/回放体验打磨成熟后,作为 **Phase 3** 提上来。
- **预估改动**:布局算法(可能要引入 dagre/elk 类)、subgraph 展开/折叠、跨 agent 的 run 关联。

## B3. 协作 / 非工程师可用 / 可分享回放

可分享的 run 回放链接、面向 PM 的视图、团队协作,且不按 seat 收费(打 LangSmith 按人头贵的痛点)。

- **价值**:LangSmith 随团队规模涨价是明确被抱怨的痛点。
- **为什么暂缓**:这是**商业化 / 产品形态**的事,不该进一个还没跑通的本地 MVP;且与"本地优先"原则需要协调(分享=某种形式的导出/托管)。
- **触发条件**:有人真的在本地版上产生了"想给同事看这次 run"的需求之后。
- **注意**:实现时不能损害"本地 / 隐私"底线 —— 优先考虑「导出单文件回放包」而非「上云」。

## B4. 框架无关(OTel ingest)

支持 LangGraph 之外的框架(CrewAI / AutoGen / 自定义),大概率通过吃 OpenTelemetry span 实现,而非逐个框架写 adapter。

- **价值**:扩大受众。
- **为什么暂缓**:现在就通用会被迫退回"一堆 OTel span"的最低公共分母 —— 正是 Langfuse/Phoenix/Laminar 已做得很好、最拥挤的赛道,会丢掉"深度"这个唯一能赢的点。
- **架构已留口**:事件协议本身框架无关,LangGraph 逻辑全在 `langgraph_visualizer/` adapter 里。未来加新框架=加新 adapter,不用动后端/前端。
- **触发条件**:LangGraph 这条验证成功、有人用了之后;且优先做"通用 OTel adapter"而非逐框架适配。

---

## 决策原则(为什么这些被留在 backlog)

进 MVP 的标准:**看得见、改得动、单人能做、且强化"本地+隐私+根因"护城河**。
不满足的(独立产品 / 高技术壁垒 / 商业化形态 / 通用化)一律先放这里。
